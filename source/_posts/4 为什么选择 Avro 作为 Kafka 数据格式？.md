# 4 为什么选择 Avro 作为 Kafka 数据格式？

如果你刚开始使用 Kafka，首先需要做的一件事就是选择数据格式。最重要的是在使用过程中保持一致性。无论是 XML、JSON 还是 ASN.1，只要始终如一地使用一个格式，都比随意混用不同格式要好得多。

如果你从零开始使用 Kafka，则可以选择自己偏好的格式。那么，哪个格式最好呢？选择时需要考虑多个标准：效率、易用性、编程语言支持等。在我们的实践中，我们发现 [Apache Avro](http://avro.apache.org/docs/current/) 是用于流数据的一个不错的选择。

------

### 什么是 Avro？

Avro 是一个开源的数据序列化系统，它帮助不同系统、编程语言和处理框架之间进行数据交换。Avro 定义了数据的二进制格式，并且能够将数据映射到你所选的编程语言中。

------

### 为什么在 Kafka 中使用 Avro？

[Confluent 平台](http://docs.confluent.io/current/) 支持任意你喜欢的数据格式，但我们为 Avro 提供了一些[特殊功能](http://docs.confluent.io/current/schema-registry/docs/)，因为它很受欢迎。在下文中，我们将探讨使用 Avro 的一些原因。

Avro 拥有类似 JSON 的数据模型，但可以表现为 JSON 或者紧凑的二进制形式。它还提供了一个复杂的模式描述语言，用于定义数据的结构。

我们认为 Avro 是理想选择的几个主要原因包括：

1. 与 JSON 之间可以直接映射。
2. 格式紧凑。JSON 在每条记录中都重复字段名，这使得它在高流量场景下效率低下。
3. 数据读取速度快。
4. 支持多种编程语言，并且可以生成 Java 对象来简化事件数据处理，但不强制代码生成，这使得开发者可以为任意数据流编写通用工具。
5. 使用 JSON 定义的丰富且可扩展的模式语言。
6. 提供最佳的数据演进兼容性支持。

虽然看起来这些元数据管理并不重要，但实际上它在组织层面上保证数据质量和易用性方面起着关键作用。

------

### Avro 的模式示例

Avro 的一个关键特性是可以为数据定义模式。例如，一个代表产品销售事件的数据可能如下所示：

```
json复制代码{
  "time": 1424849130111,
  "customer_id": 1234,
  "product_id": 5678,
  "quantity": 3,
  "payment_type": "mastercard"
}
```

它的模式定义如下：

```
json复制代码{
  "type": "record",
  "doc": "此事件记录了一次产品销售",
  "name": "ProductSaleEvent",
  "fields": [
    {"name": "time", "type": "long", "doc": "购买时间"},
    {"name": "customer_id", "type": "long", "doc": "顾客ID"},
    {"name": "product_id", "type": "long", "doc": "产品ID"},
    {"name": "quantity", "type": "int", "doc": "数量"},
    {
      "name": "payment",
      "type": {
        "type": "enum",
        "name": "payment_types",
        "symbols": ["cash", "mastercard", "visa"]
      },
      "doc": "支付方式"
    }
  ]
}
```

实际事件可能会包含更多字段，并且会有更详细的注释说明，但这展示了其基本格式。

### Kafka 与模式的关系

你可以为每个 Kafka 主题关联一个模式，这类似于关系数据库表的模式定义。模式规定了写入主题的数据要求，也为读取数据提供了解释指引。

模式的作用包括：

1. **字段校验**：确保数据流中的事件包含必要字段，并且字段类型正确。
2. **文档化**：通过模式中的 `doc` 字段描述每个字段的含义，为事件使用提供文档支持。
3. **数据验证**：避免错误数据流入主题，只允许符合模式的数据进入。

当数据规模扩大、依赖系统增多时，模式的重要性会愈加显现。没有模式的情况下，各系统之间的协作会非常脆弱且容易出错。

------

### 为什么需要模式？

虽然大数据领域常说“无结构数据”是趋势，但我们认为**正确使用模式会极大地提升数据质量和敏捷性**。模式的坏名声源于历史上关系数据库在模式变更上的复杂性，以及现代分布式系统在语义建模上的不成熟。

#### **稳健性**

在数据流系统中，各个应用是解耦的，即生产者生成事件数据，而无需了解哪些消费者会订阅这些数据。然而，这种解耦也带来了挑战：如何确保数据的正确性？

由于测试所有生产者与消费者的组合并不可行，缺乏模式会导致数据不一致。例如，有些生产者可能会错误地省略关键字段或不一致地使用字符串常量。

#### **清晰性与语义**

没有统一的模式，数据的实际含义容易在不同应用之间产生误解。我们发现，许多公司会将数据字段的说明写在 wiki 或邮件中，但这些非正式文档经常会被遗忘或与实际数据脱节。通过模式中的 `doc` 字段，可以为每个字段提供权威定义，避免这种情况。

#### **兼容性**

随着业务和代码的发展，数据格式也需要演变。模式管理为数据格式的变化提供了规范。增加字段通常是安全的，但删除字段或修改字段类型可能会导致兼容性问题。模式管理允许你提前评估这些更改的影响，避免大规模的数据重新处理。

------

### 数据科学中的模式优势

在实际工作中，数据科学家往往花费大量时间在数据清洗和解析上，而不是进行真正的数据分析。使用模式可以减少这种低效劳动，让数据科学家更专注于他们的核心工作。

------

### 结论：为什么选择 Avro？

我们评估了多种常见格式，如 JSON、XML、Thrift 和 Protocol Buffers，最终选择了 Avro。其原因包括：

- **可读性与存储效率**：Avro 提供 JSON 表示用于可读性，同时支持二进制格式以提高存储效率。
- **精确的兼容性模型**：支持数据演进和兼容性检查。
- **良好的 Hadoop 集成**：与 Hadoop 和 Hive 等大数据系统无缝兼容。
- **广泛的编程语言支持**：方便在不同系统间使用。

------

### Avro 实践建议

- **使用枚举类型**：避免使用字符串常量，减少拼写错误带来的问题。
- **为每个字段编写注释**：即使是看似简单的字段也可能有不明显的细节。
- **避免复杂类型**：避免使用递归类型或复杂的联合类型，以确保跨系统的兼容性。
- **采用一致的命名规范**：确保不同事件中的常用字段名称一致，方便在 Hadoop 中进行数据关联。
