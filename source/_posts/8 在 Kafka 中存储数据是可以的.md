# 8 在 Kafka 中存储数据是可以的

一个人们经常问的问题是：在 Apache Kafka® 中长期存储数据是否合适？你可能知道，Kafka 存储了一份日志记录，大概如下所示：

![img](https://cdn.confluent.io/wp-content/uploads/log-of-records.png)

问题在于，你是否可以将这份日志当作文件来使用，并将其作为数据的真实来源。

显然，这**是可行的**。如果你将保留策略设置为“永久”或对某个主题启用[日志压缩](https://kafka.apache.org/documentation.html#compaction)，数据就会一直被保留。但我认为人们真正关心的并不是这是否可行，而是这是否会显得太疯狂。

**简短回答是：这并不疯狂。** 人们一直在这么做，而且 Kafka 的设计本来就支持这种用法。那么，为什么你会想这么做呢？其实有很多使用场景，下面列举几个：

1. **事件溯源（Event Sourcing）**
   如果你正在构建一个基于事件溯源的应用程序，你可能需要存储一份变更日志。理论上你可以用任何系统来存储这份日志，但 Kafka 能直接解决很多与不可变日志和基于日志的“物化视图”相关的问题。[《纽约时报》](https://www.confluent.io/blog/publishing-apache-kafka-new-york-times/) 就是这样使用 Kafka 来处理所有文章数据，并作为其 CMS 系统的核心。
2. **内存缓存的更新**
   每个应用实例中的内存缓存都可以通过 Kafka 的更新来驱动。最简单的实现方式是将 Kafka 的主题日志压缩，并让应用在重启时从偏移量 0 处开始读取，以填充缓存。
3. **流处理和重新计算**
   流处理作业会基于 Kafka 的数据流进行计算。当流处理逻辑发生变化时，通常需要重新计算结果。最简单的方式是将程序的偏移量重置为 0，以便使用新的代码重新计算。这种方法有时被称为[卡帕架构（Kappa Architecture）](https://www.oreilly.com/ideas/questioning-the-lambda-architecture)。
4. **数据库更新的捕获和分发**
   Kafka 经常被用来捕获并分发数据库更新（即“变更数据捕获”或 CDC）。消费这些数据的应用在正常运行时只需要最新的变更，但新应用通常需要一份全量快照来初始化。然而，对生产数据库进行全量导出既耗时又复杂。对包含这些变更流的 Kafka 主题启用日志压缩后，消费者可以通过重置偏移量来简单地重新加载数据。

### 所以，在 Kafka 中存储数据是疯狂的吗？

**答案是否定的**，将数据存储在 Kafka 中并不疯狂。Kafka 为这种使用方式而设计，数据会被持久化到磁盘，并经过校验和复制以实现容错。即使存储的数据量增加，也不会影响性能。在生产环境中，一些 Kafka 集群存储的数据量甚至超过 1PB。

那么，既然 Kafka 本质上是一个存储系统，为什么会让人们对数据存储如此担忧呢？

#### Kafka 与传统消息队列的区别

我认为主要原因是，Kafka 经常被描述为一个消息队列。而传统消息队列的第一法则就是：**“不要在消息队列中存储消息。”** 以下是传统消息系统无法存储消息的一些原因：

- 读取消息会将其删除。
- 随着数据量超出内存限制，消息系统的扩展性变差。
- 消息系统通常缺乏强大的复制功能（如果代理宕机，数据可能会丢失）。

这些都是传统消息系统的重大缺陷。毕竟，任何异步消息传递都需要存储消息，即使只是暂时存储几秒钟以待消费。如果系统无法可靠地存储消息，就无法真正“排队”消息。在高负载的情况下，系统中总会有未消费的消息，这意味着如果系统崩溃且没有容错存储能力，数据**必然**会丢失。因此，要做好消息传递，必须解决存储问题。

然而，Kafka 的架构与传统消息队列完全不同。它更像是一个**分布式文件系统或数据库**，主要有以下三个区别：

1. **Kafka 存储的是持久化日志**，可以反复读取，且可以无限保留。
2. **Kafka 是现代分布式系统**，支持集群运行，能够弹性扩展，并具备数据复制功能，以实现高可用性和容错。
3. **Kafka 支持实时流处理**，不仅限于逐条处理消息，而是支持高层次的数据流操作。

因此，我们认为 Kafka 与其说是消息队列，不如称之为[流平台](https://www.confluent.io/blog/stream-data-platform-1/)更合适。

### Kafka：融合存储与消息传递的最佳平台

消息系统关注的是**未来**的消息，而存储系统处理的是**过去**的记录。Kafka 的核心抽象是一种**连续的有序日志**，它将这两者结合起来——既可以处理历史数据，也可以处理未来发生的事件。这种日志就像一个结构化文件，但它不会在末尾停止，而是**逻辑上无限延续**。因此，基于日志的程序无需区分已发生的数据与将来的数据，所有数据都呈现为一个连续流。

Kafka 的这种日志类似于分布式文件系统中的文件：它会跨多台机器复制，持久化到磁盘，并支持高吞吐量的线性读写。同时，它也像消息系统，支持大量并发写入，并定义了精确的消息发布时机，以实现低延迟的变更传播。

事实上，Kafka 本身也将其日志用作存储。Kafka 会在[压缩主题](https://kafka.apache.org/documentation#compaction)中存储消费者的偏移量，并使用这些压缩主题来记录应用的处理状态。

### 作为主要存储系统的运维挑战

将 Kafka 作为主要存储系统使用确实提高了运维要求。存储系统需要高可靠性和数据完整性。我曾参与 LinkedIn 的多代分布式数据库开发，当系统被视为数据的唯一来源时，软件的正确性和运维水平的要求会显著提高。我们已经在 Kafka 的[正确性](https://www.confluent.io/blog/apache-kafka-tested/)上投入了大量努力，每天进行数百小时的分布式压力测试，并运行数千个单元测试，但仍然有改进空间。如果你要将 Kafka 用于这种场景，需要深入了解其运维，并掌握其局限性。

### Kafka 是通用数据库的未来吗？

有人问，Kafka 会不会成为一种通用数据库，取代所有其他存储系统？答案是否定的，原因如下：

1. **数据库主要用于查询**，而 Kafka 不擅长直接针对日志进行随机访问。Kafka 更适合用作日志系统，并将数据同步到各种数据库、缓存、数据湖等系统中，这些系统有各自的优缺点。
2. **不同系统针对不同存储需求进行了优化**，如分析数据库使用列式存储，搜索引擎使用倒排索引，缓存依赖内存存储。因此，单一系统难以超越所有这些优化。

Kafka 更像是数据中心的提交日志，而其他存储系统则是派生索引或视图。Kafka Streams API 的**交互式查询**功能就是一个例子，它允许应用从流数据中维护状态，并对其进行查询。这些查询并不直接针对 Kafka，而是基于 Kafka 计算并存储的物化视图。

此外，[KSQL](https://www.confluent.io/blog/ksql-open-source-streaming-sql-for-apache-kafka/) 的加入进一步降低了开发门槛，允许通过 SQL 语句实现数据流的转换和查询。

### 结语

我们的目标是让数据流和流处理成为主流开发模式，让 Kafka 成为现代企业的“**数据中枢**”。我们相信，流平台将在现代实时应用的构建中发挥重要作用。
