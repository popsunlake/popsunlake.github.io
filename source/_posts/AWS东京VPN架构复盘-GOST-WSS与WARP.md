最近我把自己在 AWS 东京区运行了两年多的自用代理节点重新梳理了一遍。这个节点平时主要用于个人设备访问海外网站，整体稳定性不错，但也有几个问题值得复盘：为什么它比早期 Shadowsocks 方案稳定，为什么晚高峰吞吐会下降，以及如果只跑一个自用节点，当前每月十几美元的 AWS 成本是否还有优化空间。

这篇文章不贴真实公网 IP、域名、用户名、密码和证书私钥，只记录架构和分析方法。

## 当前架构

从客户端配置和服务端脚本看，当前日常使用的主链路并不是传统 Shadowsocks 裸服务端，而是：

```text
Windows 应用/浏览器
    |
    v
本机代理 127.0.0.1:1080
    |
    v
Shadowsocks Windows 客户端
    |
    v
GOST Windows 插件
    |
    |  WSS/TLS 443
    v
AWS EC2 东京实例
    |
    v
GOST Linux 服务端
    |
    |  AWS 东京公网 IP 出口
    v
Google / YouTube / 其他海外网站
```

另外还有一条备用链路：

```text
Windows 客户端
    |
    |  WSS/TLS 2083
    v
AWS EC2 东京实例
    |
    v
Cloudflare WARP 本地 SOCKS5 代理
    |
    |  Cloudflare 出口 IP
    v
OpenAI / Bing / 其他对云厂商 IP 敏感的网站
```

所以它可以概括为：**Shadowsocks Windows 做客户端入口，GOST WSS 443 做主通道，AWS 东京做主出口，WARP 2083 做备用出口**。

## 每一部分如何工作

### 1. 本机代理入口

客户端机器上，浏览器和其他应用通常先把请求发给本机代理端口，例如：

```text
127.0.0.1:1080
```

这个端口由 Shadowsocks Windows 客户端提供。浏览器访问 YouTube 时，并不是直接访问目标网站，而是先把请求交给本机代理：

```text
Chrome -> 127.0.0.1:1080 -> Shadowsocks Windows
```

如果某个应用没有使用系统代理，或者没有单独配置代理，那么它的流量就不会经过这条链路。

### 2. Shadowsocks Windows 的角色

从界面看，客户端仍然使用 Shadowsocks Windows，但它更像一个图形化外壳和本地代理管理器。真正负责远端传输的是 GOST 插件。

客户端配置里可以看到类似信息：

```text
服务器端口: 443
加密方式: chacha20-ietf-poly1305
插件程序: gost-windows-amd64
插件选项: 用户名:密码
```

也就是说，Shadowsocks Windows 负责把本地流量交给 GOST 插件。它保留了 Shadowsocks 客户端易用的优点，比如系统托盘、代理模式、本地端口管理等，但跨境传输的外观已经不是传统 Shadowsocks 裸流量。

### 3. GOST Windows 插件

GOST 插件会把本地代理流量封装成远端 GOST 服务端能够识别的连接。当前主线路使用的是 443 端口，传输形态是 WSS，也就是 WebSocket over TLS。

一次访问可以简化成：

```text
本地请求:
访问 https://www.youtube.com

封装后:
通过 WSS/TLS 连接到 AWS 东京节点的 443 端口
携带 GOST 用户认证信息
```

这一步的价值在于，链路外观看起来更接近普通 HTTPS/WebSocket 连接，而不是特征更明显的传统代理协议。

### 4. AWS EC2 东京实例

AWS 东京实例承担两个职责：

1. 接收本地客户端发来的加密代理连接。
2. 从东京区域向目标网站发起真实访问。

从目标网站视角看，请求来源不是国内网络，而是 AWS 东京公网 IP。访问链路变成：

```text
国内客户端 -> AWS 东京 -> 目标网站
```

这个设计的好处是链路短、可控性强。AWS 东京区域的可用性也比较稳定，实际运行两年多没有频繁故障，说明这个 IP、域名和流量模式目前处于相对健康的状态。

### 5. GOST Linux 服务端

服务端脚本会在机器上安装 GOST，并生成类似这样的配置：

```json
{
  "ServeNodes": [
    "wss://:443?secrets=/opt/gost/secrets.txt&cert=/opt/gost/ca.pem&key=/opt/gost/key.pem"
  ]
}
```

这表示服务端监听 443 端口，使用 WSS 协议，并通过 `secrets.txt` 做用户认证。证书和私钥用于 TLS 连接。

当客户端访问 Google 时，服务端做的事情大致是：

```text
1. 接收来自客户端的 WSS 连接。
2. 校验用户名和密码。
3. 解析客户端真正要访问的目标地址。
4. 从 AWS 东京实例向目标网站建立连接。
5. 把目标网站响应再通过 WSS 回传给客户端。
```

### 6. 443 直连线路

443 是当前日常使用的主线路。它的出口是 AWS 东京公网 IP，链路相对短：

```text
国内客户端 -> AWS 东京 -> Google / YouTube
```

这条线路适合大多数普通访问场景，例如 Google、YouTube、网页浏览和下载。它没有额外绕 WARP，因此通常比 2083 备用线路更快。

### 7. 2083 + Cloudflare WARP 备用线路

脚本还配置了一个 2083 端口。这个端口收到流量后，不直接从 AWS 出口访问目标网站，而是转发到本机的 Cloudflare WARP SOCKS5 代理。

简化链路如下：

```text
国内客户端 -> AWS 东京 -> Cloudflare WARP -> 目标网站
```

这条线路的主要价值不是速度，而是出口 IP 变化。某些服务对云厂商 IP 比较敏感，直接使用 AWS 出口可能触发风控；通过 WARP 出口，有时可用性会更好。

但这条线路也有明显代价：多了一跳 Cloudflare WARP，延迟、丢包和拥塞都会被放大。因此它更适合作为 OpenAI、Bing 或特定网站的备用线路，不适合作为所有流量的默认线路。

## 为什么这个方案比较稳定

第一，主通道跑在 443 端口上。443 是 HTTPS 默认端口，在绝大多数网络环境中都属于高频正常流量。

第二，传输层使用 WSS/TLS。相比裸 Shadowsocks，WSS over TLS 的外观更接近常见 WebSocket/HTTPS 流量，特征暴露更少。

第三，AWS 东京节点质量相对稳定。很多低价 VPS 的 IP 段质量、路由和可用性波动更大，而 AWS 东京作为主流云区域，基础设施稳定性更好。

第四，使用规模较小。单人或少量设备自用时，流量模式更自然，不容易形成异常的高并发、多用户、大流量特征。

第五，有备用出口。443 直连 AWS，2083 可以转 WARP。即使某些服务对 AWS IP 不友好，也有备用路径。

## 为什么会慢，尤其晚高峰

当前配置已经使用 443 直出 AWS，因此如果日常速度慢，主要原因未必是 WARP。更可能来自以下几类瓶颈。

### 跨境线路拥塞

晚高峰变慢通常首先怀疑国内运营商到日本方向的跨境链路拥塞。即使 AWS 东京本身没有问题，国内到东京的路径也可能在晚上出现丢包、绕路或带宽收缩。

### WSS/TLS 协议开销

WSS over TLS 的稳定性较好，但它不是最低开销的传输方式。相比一些 UDP/QUIC 类方案，WSS 在延迟和吞吐上可能吃亏，尤其是单连接大流量场景。

### 实例规格限制

如果 EC2 实例规格较小，CPU 加密性能、网络性能档位、突发积分等都可能影响吞吐。代理场景看似轻量，但高码率视频、下载和多设备并发时，CPU 和网络都可能成为瓶颈。

### WARP 备用线路的额外跳转

如果客户端切到 2083，那么流量会经过 WARP。这个链路更长：

```text
客户端 -> AWS 东京 -> WARP -> 目标网站
```

它适合解决可用性问题，但通常不适合追求最高吞吐。

## 成本结构

当前 AWS 账单大致可以拆成三部分：

```text
EC2 实例: 主要计算资源费用
VPC / 公网 IPv4: 公网 IPv4 按小时收费
EBS: 系统盘费用
```

对单机自用代理来说，EC2 的成本不一定最优。因为即使实例缩小，公网 IPv4 和磁盘费用仍然存在。更适合的 AWS 形态可能是 Lightsail：它把实例、磁盘、公网 IPv4 和一定流量包打在一个固定套餐里，账单更简单，也更接近这类单机服务的使用方式。

不过成本优化不能只看月费。这个节点已经稳定运行两年多，说明当前 IP 和线路有实际价值。直接迁到最便宜的 VPS，可能省了几美元，但换来更高的 IP 风险和线路波动。

## 当前方案的优点

1. 稳定性强于传统 Shadowsocks 裸服务端。
2. 443 端口自然，网络兼容性好。
3. WSS/TLS 外观更接近常规 Web 流量。
4. AWS 东京区域稳定性较好。
5. 有 AWS 直连和 WARP 备用两条出口路径。
6. 已经经过两年多实际验证。
7. 服务端已启用 BBR，对长距离 TCP 传输有帮助。

## 当前方案的缺点

1. 成本偏高。EC2、IPv4、EBS 加起来对单人自用代理不算便宜。
2. WSS/TLS 有额外协议开销，极限吞吐不一定最好。
3. 晚高峰仍然受跨境线路影响，服务器稳定不等于链路永远稳定。
4. 2083 + WARP 多一跳，适合解锁，不适合默认高速访问。
5. 脚本里如果硬编码证书、私钥和密码，维护和安全性较差。
6. `iptables -I` 添加的规则可能不持久，重启后需要确认防火墙状态。
7. GOST 版本固定较旧，长期需要评估升级兼容性。
8. Shadowsocks 客户端 + GOST 插件 + GOST 服务端 + WARP 的组合排障成本较高。

## 优化思路

我更倾向于按风险从低到高优化。

第一步，保持现有 AWS EC2 不动，先做基准测试。分别记录非高峰和晚高峰的延迟、丢包、YouTube 播放档位、下载速度、CPU 使用率和内存使用率。

第二步，明确 443 和 2083 的使用边界。普通 Google、YouTube、网页和下载优先走 443；只有 OpenAI、Bing 或特定风控网站再切 2083 + WARP。

第三步，检查实例规格。如果当前实例规格偏大，可以评估缩容；如果迁到 ARM Graviton，需要把服务端 GOST 二进制从 amd64 换成 arm64。

第四步，并行测试 Lightsail 东京小套餐。不要直接替换当前稳定节点，而是新建一个节点，复用同样的 GOST WSS 443 架构，连续观察 7 到 14 天。

第五步，再考虑第三方 VPS。Vultr、Oracle Cloud 等方案可能更便宜，但 IP 段质量和线路波动需要实测，不建议直接替换一个已经稳定两年的 AWS 东京节点。

## 总结

这个节点稳定的核心并不是 Shadowsocks 本身，而是 GOST WSS 443、AWS 东京线路和低强度自用流量模式的组合。Shadowsocks Windows 在当前架构中更多承担客户端外壳的角色，真正的传输通道由 GOST 完成。

如果目标是稳定优先，当前方案值得保留。如果目标是降本，优先考虑 AWS Lightsail 东京并行验证，而不是直接换到最低价 VPS。如果目标是提升吞吐，应该先区分 443 直连和 2083 WARP，再通过晚高峰对比测试判断瓶颈是实例规格、协议开销，还是跨境线路拥塞。

最稳妥的策略是：**当前 EC2 保留，新节点并行跑，验证通过再迁移**。
