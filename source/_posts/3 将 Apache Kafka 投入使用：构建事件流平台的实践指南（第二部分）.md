# 3 将 Apache Kafka 投入使用：构建事件流平台的实践指南（第二部分）

这是我们关于流数据和 Apache Kafka 指南的第二部分。在[第一部分](http://blog.confluent.io/stream-data-platform-1)中，我介绍了实时数据流的用途，并解释了事件流平台的概念。本指南的剩余部分将提供如何在您的组织中构建事件流平台的具体建议。

这些建议基于我们在 LinkedIn 构建和实施 Kafka 的经验，并在各类数据类型和系统中进行推广。此外，我们还基于多年与企业合作构建 Kafka 驱动的事件流平台的实践，总结了相关经验。

这是一份**动态文档**，随着我们学习到新的技术或新工具的发布，我会不断更新内容。

## 入门

本指南中的许多建议适用于扩展到数百或数千个规范化的数据流。当然，没人一开始就做到这一步。通常情况下，您可以从一两个试点应用程序入手，尤其是那些对系统扩展性要求较高的应用场景，因为传统系统往往不太适合这种需求。即便是这种有限部署的情况，本指南中的技术也能帮助您从一开始就采用良好的实践，为未来的扩展打下坚实基础。

**从小规模开始**是明智的，这样可以在实践中了解什么有效、什么无效，为更广泛的应用做好准备。

## 推荐建议

我将首先给出关于流数据和 Kafka 的一般性建议，然后讨论不同类型事件流的具体情况。

### 1. 限制集群数量

在早期的试验阶段，不同部门在采用 Kafka 时，往往会形成多个 Kafka 集群。然而，数据管理的一个核心目标是集中管理数据流，即将所有数据流集中到一个中央存储库中，这类似于数据仓库的推荐实践，其目标是将数据集中到一个中央仓库，以简化操作并促进多数据源的整合。

同样，我们发现，将流数据存储在尽可能少的 Kafka 集群中，可以极大地简化系统架构。这意味着数据消费者的集成点更少，系统的运维负担降低，新增应用的成本也更低，还可以更清晰地理解数据流向。

当然，集群数量最少并不一定意味着**只有一个集群**。以下是使用多个集群的常见原因：

1. **保持本地数据中心的活动本地化**。我们建议所有应用程序连接到其本地数据中心的集群，并通过数据中心之间的镜像功能同步数据。
2. **出于安全考虑**。尽管 Kafka 具备[强大的安全功能](http://docs.confluent.io/current/kafka/security.html)，有时还是需要物理上隔离某些数据。
3. **SLA 控制**。虽然 Kafka 使用[配额](https://kafka.apache.org/documentation/#design_quotas)来隔离应用程序，但物理上分离集群仍有助于限制任何操作事故的影响范围。

早期 Kafka 版本缺乏强大的安全性和配额控制，鼓励了通过多个集群实现隔离。但现在利用内置功能通常是更好的选择。

### 2. 选择统一的数据格式

Kafka 本身对事件数据格式没有严格要求，您可以使用 XML、JSON 或 Avro 等格式。我们的理念是，数据基础设施系统不应强制选择某种数据格式，这应由组织内部决定。

然而，**组织应统一采用一种格式**。选择单一数据格式有助于简化集成，不仅是因为所有数据都在 Kafka 系统中，还因为所有数据格式一致，遵循相同的规范。如果不同应用程序使用不同格式（如 JSON、XML 和 CSV），则每个系统都需要处理和转换这些格式，这将增加系统的复杂性。

一个朋友的类比可以很好地解释数据格式的重要性：Unix 命令行工具之间的成功集成依赖于**换行符分隔的 ASCII 文本**标准。这些工具尽管由不同开发者在不同时间编写，但都可以通过‘|’管道符将数据从一个命令传递到另一个命令。如果每个工具使用自己的格式，那么这些工具链将变得无法使用。

尽管换行符分隔文本在今天可能已经不够用了，但这个例子说明了格式标准化的重要性。选择一种格式，并确保所有工具和集成都使用该格式，是实现事件流平台的关键。

### 数学中的简化法则

通过**限制集群数量**和**标准化数据格式**，数据流的复杂性可以显著降低。

集中式事件流平台将所有系统连接到一个中央平台，简化了数据流模型。如图所示：

- **传统集成模型**：

  ![传统模型](https://cdn.confluent.io/wp-content/uploads/data-systems-300x298.png)

- **事件流平台模型**：

  ![事件流平台](https://cdn.confluent.io/wp-content/uploads/data-systems-sdp-279x300.png)

在传统模型中，N 个系统之间需要构建 N^2 条数据管道，而在事件流平台模型中，每个系统只需与平台建立输入和输出连接。这不仅减少了管道数量，还减少了系统之间格式转换的需求。

### 3. 优选 Avro 作为数据格式

任何格式（XML、JSON、ASN.1 等），只要一致使用，都比随意选择的混合格式要好。然而，如果您从头开始使用 Kafka，我们推荐[Apache Avro](http://avro.apache.org/docs/current/)作为标准格式。Avro 在兼容性和工具支持方面有诸多优势，具体可以参考[这里](https://www.confluent.io/blog/avro-kafka-data/)。

### 4. 谨慎选择客户端

Kafka 提供了丰富的[客户端](https://cwiki.apache.org/confluence/display/KAFKA/Clients)支持。Java 客户端由 Apache 项目开发，性能优秀。而对于非 Java 客户端，我们推荐基于 C 库 **librdkafka** 的客户端。这是一个经过大量实战验证的高性能客户端，并支持多种语言，如 Python、Go 和 C#。

### 5. 事件优于命令

Kafka 的数据模型本质上是发布/订阅模式，因此采用**事件**而非**命令**的消息结构更为合适。事件表示某个行为已经发生，而命令是向某个系统发出具体指令。

例如，当用户注册后，我们可以：

1. 写入“发送邮件”命令；
2. 写入“用户注册事件”。

第二种方式更具通用性，因为除了发送邮件外，“用户注册事件”还可用于分析和其他服务触发。

### 6. 使用 Kafka Connect 连接系统

Kafka 提供的 [Kafka Connect](http://docs.confluent.io/current/connect/) 框架，可以方便地将外部系统与 Kafka 集成。它解决了大规模系统集成中的扩展性和容错问题，并通过 REST API 管理多个连接。

![Kafka Connect](https://cdn.confluent.io/wp-content/uploads/connect-1-300x165.png)

### 7. 理解流处理生态

Kafka 0.10 版本引入了[Streams API](https://www.confluent.io/blog/introducing-kafka-streams-stream-processing-made-simple/)，使流处理变得更加简单。该 API 将状态管理和处理逻辑分布在应用实例之间，实现动态扩展和故障恢复。

![Streams 应用](https://cdn.confluent.io/wp-content/uploads/streams-280x300.png)
